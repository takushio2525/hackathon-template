# コントリビューションガイド

このリポジトリで開発するときのルール集。チーム全員が守ること。

## ブランチ戦略

- `main` ブランチへ直接 push は **禁止**
- 機能追加・バグ修正は `feature/<short-description>` or `fix/<short-description>` の feature branch で作業
- 作業完了後 `main` へ向けて Pull Request を作成
- PR は最低 **1名** のレビュアー承認必須
- **共通モジュール（`firmware/common/lib/`）を変更する PR は最低2名のレビュアー承認必須**

## コミットメッセージ規約

`[種別] 変更内容の概要` の形式とする。

| 種別 | 用途 |
|------|------|
| `[機能追加]` | 新機能の実装 |
| `[修正]` | バグ修正 |
| `[改善]` | 既存機能の改善 |
| `[リファクタ]` | コード整理（機能変更なし） |
| `[ドキュメント]` | ドキュメントのみの変更 |
| `[スタイル]` | 見た目・UI の変更 |

- 機能単位で細かく分割する（無関係な変更を1コミットにまとめない）
- コード変更とドキュメント更新は同一コミットに含める

## レビュー観点

PR レビュー時に以下をチェックすること。

### コード全般
- 命名規則に従っているか（`CLAUDE.md` / `Embedded-Module-Architecture` 参照）
- 不要なコメントアウト・デバッグ print が残っていないか
- 型・配列サイズが適切か

### モジュール（`IModule` 準拠）
- 3フェーズ実行モデルに違反していないか（入力 → ロジック → 出力）
- ピン番号がハードコードされていないか（`Config` 経由でアクセスしているか）
- `delay()` を使っていないか（`ModuleTimer` 使用）
- `SystemData` を直接メンバに保持していないか

### ドキュメント
- README の更新が必要な変更ではないか
- 設計判断が大きい場合は `docs/decisions/` に ADR を追加したか

## PR テンプレート

`.github/PULL_REQUEST_TEMPLATE.md` に沿って記載すること。

## Issue で作業を管理する

- 分担タスクは Issue を立てて管理
- `.github/ISSUE_TEMPLATE/task.md` を使う
- PR から関連 Issue へ `Closes #NN` で紐付け

## 共通モジュール担当表

`firmware/common/lib/README.md` を参照。担当者が決まり次第，そこに追記する。
